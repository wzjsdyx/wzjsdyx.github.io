IC设计流程之FT测试

发表于 2024-10-14 更新于 2024-10-17 阅读次数： Valine：

补充FT pattern

AC7843需要对ATPG AU fault补充function pattern

AU的fault都是ATPG在test_mode下面受自身或者mode限制无法达到的fault

FT打不到的fault，则考虑由SFT去测试

FT和SFT的区别：

SFT pattern需要将bin给写入到eFlash中去执行；

STEP1：fault list分析

基本概念理解

==》fault？AU fault？

Tool能够检测到fault，如果某一个fault是可控制并且可观测的，那么ATPG的test pattern可以覆盖这个可能的故障点（检测出stuck错误，从而筛除）

可控制：

某一个CELL的输入信号需要是能够toggle的；

电信的错误是：一个或门，其中一个输入逻辑是test scan mode，那么ATPG下，其输出结果永远是1，就无法检测到stuck 0的错误；

可观测：

摸一个CELL的输出可以传递到另外一个CELL

典型的错误是：mux的选择信号被test scan mode控制；另外一路就无法传播到后一级

必须同时满足可控制和可观测，否则就为AU(ATPG Untestable) fault，需要DE来补function pattern

方法学

==》分析report上 fault没有被ATPG覆盖的原因

report报告：

1、report上很多的fault可以不用去看

  • Netlist 里面被tie 0 或者tie1, dont care
  • sram的output fault在目前的atpg里面是不测试的，但是会被mbist cover，dont care

2、需要补FT function pattern

  • Analog output signal，这部分的fault点因为没有办法控制，所以没能被atpg cover，需要补充funciton pattern
  • Pinmux内部的很多的fault是由于scan_mode. Test_mode, scan_en 等信号被固定了input . output，需要补一下其他mode的FT来cover.
  • Gpio相关的和Pinmux的fault点类似，也是mode选择固定的了PAD的一些控制信号，需要补FT来cover其他mode
  • Flash output related, 这部分CP和SFT都会cover，dont care
  • Reset在scan的时候被换到Scan_reset，normal reset需要update FT cover
  • Spm内部出的analog input 信号，在test mode的时候是被tie固定值的，update FT去cover别的case.

// Analysis summary:
// The fault site cannot be set to 1.

报告解释

  1. 命令：
    • analyze_fault -display -stuck_at 0 /MTCMOS_wrap/protsys/u_mpu_top_0/i_mpu_core_h/U740/ZN
      这条命令用于分析指定故障，即**/MTCMOS_wrap/protsys/u_mpu_top_0/i_mpu_core_h/U740单元输出端ZN卡0**的故障状态。
  2. 故障描述：
    • Fault analysis for /MTCMOS_wrap/protsys/u_mpu_top_0/i_mpu_core_h/U740 (1439325) output ZN stuck at 0
      • 分析的故障是模块/MTCMOS_wrap/protsys/u_mpu_top_0/i_mpu_core_h/U740中，输出端ZN被卡住为0的情况（stuck-at-0）。括号中的1439325表示U740单元在工具中的内部ID号。
  3. 故障列表状态：
    • Fault is not currently in fault list.
      • 这个故障并不在当前的故障列表中。这通常意味着该故障还没有被列为要测试的目标，或者已经被判定为不可测试。
  4. 顺序深度（Sequential Depth）：
    • Fault site sequential depth: Control_0 = 0, Control_1 = 255, Observe = 0.
      • Control_0 = 0：将故障点设为0的控制深度为0，表示可以直接将ZN置为0。
      • Control_1 = 255：将故障点设为1的控制深度为255，表示需要经过255个时钟周期（状态转换）才能将ZN设为1。
      • Observe = 0：观察点深度为0，说明可以在不需要多余的时钟周期下直接观察到故障的影响。
      • 这些值表示该故障点需要较长的时钟周期去控制ZN输出为1，这会影响测试覆盖。
  5. 激活故障的分析：
    • The fault site failed to be set to 1 (ATPG status = atpg_untestable).
      • 分析表明ZN端无法被设为1。工具无法找到能让故障节点从0切换到1的有效向量，因此被判定为不可测试（atpg_untestable）。
  6. 分析总结：
    • The fault site cannot be set to 1.
      • 总结说明ZN卡0故障无法被激活为1，也就是说，该故障被认为是顺序不可测试的故障（AU.SEQ）。这种情况下，ZN端在卡0的状态下无法通过输入控制实现1的状态，从而不能通过测试来检测到该故障。

针对AU.SEQ错误类型的说明：

假设我们有一个简单的时序电路，包含两个触发器（Flip-Flops），FF1 和 FF2，连接到一个与非门（NAND gate）。这个与非门的输出连接到FF2的输入。ZN是与非门的输出端。

假设的情况：

  1. ZN 卡0故障：在正常情况下，与非门的输出 ZN 会根据 FF1 和 FF2 的状态进行逻辑运算输出。但是，由于这个卡0故障，ZN被固定为逻辑0，即无论输入什么值，它的输出永远是0。
  2. 激活故障的挑战：
    • 为了检测 ZN 的卡0故障，需要找到一种方法让 ZN 理论上应输出1，但因为卡0故障实际输出为0。这样可以通过观察 FF2 的输出来检测到错误。
    • 根据顺序深度控制的值，报告指出需要 255 个时钟周期 才能让 ZN 理论上输出1。

测试和不可测的原因

在这个例子中，如果要让 ZN 应输出1，可能需要：

  • 设定 FF1 和 FF2 在特定的状态。这需要多个时钟周期去设置 FF1 和 FF2 的输入，以便达到这种组合。
  • 如果要达到 ZN = 1 的情况，需要让 FF1 和 FF2 在一个特定时序下相互作用，但这可能非常复杂，或者测试工具无法生成这样的序列（如需要 255 个时钟周期的控制），导致测试覆盖不足。

为什么测试工具会标记为不可测试？

因为在实际测试中，Tessent工具在设定和观测时序电路的特定状态时，可能会因为状态复杂性和循环数超出工具极限而无法设定 ZN 为1。因此，它标记这个故障为不可测试。
