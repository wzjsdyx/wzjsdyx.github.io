康奈尔笔记法

发表于 2025-04-30 Valine：

笔记的作用是对信息进行二次处理，帮助记忆理解；

step1：根据课程内容做笔记，要求快速、简易、有效、易辨识；

字体可以使用：印刷字体；

电报式写作：简写；

事无巨细写下来的不是合格的学习者；笔记需要注重观点，而不是文字，得其意，忘其形

一般情况下，尽量避免使用大纲形式，因为他会迫使自己使用高度规范的模式去填充内容，失去了自己的主观思考；

使用字母和符号来记笔记，但别让他束缚，例如用1,2,3的顺序罗列，摒弃了一些看似无法归类的闪光点；

Note:

文字干货 可以不用大段大段的抄录，可以直接说明是哪本书的哪个章节，哪一页；

step2：在休息间隙或者当天晚些时候，花时间通读你的笔记，并进行补全或者修正；

step3：找出课堂或者演讲中的主要观点，并在左栏列出对应的问题；

step4：盖住笔记，看着左侧的问题，尝试复述要点；

step5：在底部边栏为整个课堂或者演讲内容写下总结；

step6：重新回溯要点，注意全局而非片面的事实或者观点；

step7：开始进入反思过程，做一个更好的思考者；

工艺库-对比

发表于 2025-04-29 分类于 工艺库 Valine：

先说几个容易混淆的点：

同种工艺下，LVT/RVT/HVTPPA对比

  • 标准单元库PPA对比

Performance：因为LVT的阈值电压更低，因此速度更快；

Power：同样，因为LVT的阈值电压更低，动态功耗和静态功耗都最高

面积：基本一样

  • 整体chip的PPA对比

LVT比HVT其实面积上差不多，主要是LVT的驱动能力更强，一个LVT的Cell可以替代多个RVT或者HVT的Cell

T28 VS T40工艺 PPA对比

  • Performance：T28速度更快；

  • Power：T28的动态功耗更低；T28的静态功耗更高；

  • Area：T28面积更小

📚 TSMC 28nm vs 40nm 工艺在 PPA（功耗 Power / 性能 Performance / 面积 Area） 上的整体对比

性能（Performance）：
28nm比40nm快30%-50%。

功耗（Power）：

  • 动态功耗：28nm比40nm低约25%-40%。
  • 静态功耗：28nm反而比40nm更高（如果选高性能工艺，比如HPM），因为漏电增加。

面积（Area）：
28nm比40nm小约45%。

那么静态功耗和动态功耗是一个数量级吗？如果用先进工艺的话，整体的功耗会降低吗？

针对这个问题，其实需要看应用场景是性能更重要还是功耗更重要？

例如汽车芯片有侧重性能的，有侧重功耗的；

针对侧重功耗的场景： 汽车芯片：NXP S32K1系列MCU（TSMC 40

  • 运行时功耗大约100~300mW（动态主导）
  • 深睡眠时功耗小于150µW（几乎全是静态漏电）

👉 为什么S32K1用40nm？因为：

  • 汽车要求长时间待机，极低静态功耗。
  • 40nm工艺静态漏电比28nm低得多，而且工艺成熟可靠。

杰发科技 7840：

深度睡眠功耗（standby），150uW

run mode功耗：200mW

因此这种超低功耗的场景，40nm比28nm更好；

T40 VS S40工艺PPA对比

  • SRAM Macro：其实是S40的速度更快，但是功耗更高；

  • stand cell来说：

Performance：SMIC的速度更慢些；

Power：功耗更低；

Area：面积更小；

MCU信息安全初识

发表于 2025-04-25 更新于 2025-05-07 分类于 车规MCU Valine：

✅ 二、芯片信息安全（Information Security / Cybersecurity）

🌟 定义：

指的是芯片能否保护内部信息不被非法访问、修改或盗取。

📌 关键点：

  • 防止“被攻击”
  • 基于 ISO/SAE 21434 或 Common Criteria（CC EAL）
  • 典型保护机制：加密引擎、密钥存储、防篡改、访问控制、抗侧信道攻击

🔐 芯片中的信息安全机制（Information Security）

安全需求 实现方式 原因
密钥管理 PUF + Secure Key Storage 密钥不可被导出，且每颗芯片唯一
软件保护 Secure Boot + 签名校验 防止恶意固件或OTA伪装攻击
数据加密 AES/SM4 引擎 + 真随机数生成器 通信数据加密，保障机密性/完整性
访问控制 MPU（Memory Protection Unit） 防止非法访问安全区域
抗侧信道攻击 特殊加密硬件（如电流掩蔽设计） 防止攻击者通过电磁/功耗分析窃密

📎 举例：

车载网关芯片内存储了 OTA 升级的加密密钥，攻击者试图通过物理接触或软件漏洞读取密钥。如果芯片有信息安全机制，比如 PUF（物理不可克隆函数）或加密认证，攻击就无法成功。

🧩总结对比表：功能安全和信息安全

项目 功能安全（Functional Safety） 信息安全（Information Security）
目标 避免因故障导致的危险 防止非法访问/攻击
面向问题 随机硬件失效、系统错误 黑客攻击、数据泄露、未授权访问
典型标准 ISO 26262 ISO/SAE 21434、CC、FIPS 140-3
常见机制 ECC、双核锁步、监控模块 加密引擎、PUF、Secure Boot
应用场景 刹车、气囊、电池管理系统等 车联网通信、OTA升级、电子支付等

🔐 Secure Boot 的作用：

功能作用 说明
防止恶意固件 攻击者想烧录自己的病毒固件？Secure Boot不让它运行。
防止OTA篡改 OTA 升级包必须签名合法，否则无法更新。
建立信任链 从 boot ROM → bootloader → OS → 应用，全链条验证。
硬件绑定信任 使用芯片内的私钥（如 PUF 生成）验证签名，不能伪造。

📦 实际应用场景举例

✅ 场景一：汽车 OTA 升级防护你开着汽车，收到一次空中升级（OTA）通知。攻击者若拦截并替换升级包，想注入后门代码怎么办？

Secure Boot 流程如下：

  1. 升级包由厂商签名（如使用 ECC 私钥签名）
  2. 芯片内的 bootloader 用硬编码或加密存储的公钥验证签名
  3. 验证成功 → 允许烧录并启动新固件
    验证失败 → 拒绝启动，维持原状态或进入安全模式

✅ 安全效果：攻击者即使替换了升级内容，也无法通过签名校验，车不会运行伪装程序。

MCU功能安全初识

发表于 2025-04-25 分类于 车规MCU Valine：

✅ 一、芯片功能安全（Functional Safety）

🌟 定义：

指的是芯片在发生故障时，能否检测并控制故障，防止产生危险的后果。

📌 关键点：

  • 关注“失效是否被检测和控制”
  • 基于 ISO 26262 标准（车规芯片）
  • 通常涉及：冗余、电压监测、故障检测机制（BIST、ECC、WDT）

🔧 芯片中的功能安全机制（Functional Safety）

功能模块 安全设计 原因
CPU 双核锁步（Lockstep） 保证指令执行正确，有错误可检测
电源监测 电压/电流异常监测 防止供电异常造成芯片误动作
存储器 ECC（Error Correction Code） 自动纠正 SRAM/Flash 的 bit 错误
时钟模块 时钟监控器 + 看门狗 防止振荡器停止或异常时芯片卡死

📎 举例：

一个车载 MCU 控制刹车系统，如果某个电路发生故障（比如时钟停止工作），芯片能快速检测到并进入安全模式（例如紧急制动），避免事故发生。

🧩总结对比表：功能安全和信息安全

项目 功能安全（Functional Safety） 信息安全（Information Security）
目标 避免因故障导致的危险 防止非法访问/攻击
面向问题 随机硬件失效、系统错误 黑客攻击、数据泄露、未授权访问
典型标准 ISO 26262 ISO/SAE 21434、CC、FIPS 140-3
常见机制 ECC、双核锁步、监控模块 加密引擎、PUF、Secure Boot
应用场景 刹车、气囊、电池管理系统等 车联网通信、OTA升级、电子支付等

🧱 4. EVITA 安全等级（Light / Medium / Full）

定义：EVITA（E-safety Vehicle Intrusion protected Applications）是一个欧盟资助的项目，旨在为汽车电子系统建立可信的硬件支持信息安全架构。

安全等级：

  • Light：提供基本的安全功能，如对称加密和密钥存储。
  • Medium：在 Light 的基础上增加了防篡改保护和更强的密钥管理功能。
  • Full：提供最全面的安全功能，包括对称和非对称加密、完整的密钥管理、防篡改保护和安全启动等。
