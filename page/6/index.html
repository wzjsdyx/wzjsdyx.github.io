CoreMark简介

CoreMark是更加复杂的benchmark程序用于衡量现代处理的性能；

CoreMark是在2009年开发，非常容易获取以及非常容易移植；其开发的初衷就是为了替代Dhrystone benckmark。

Dhrystone是在1984年开发，在很多方面都存在问题，CoreMark可以进行弥补，具体如下：

工作负载的现实性

Dhrystone 是一个合成基准测试（synthetic benchmark）：

它通过执行一系列简单的整数运算和字符串操作来测试处理器的性能；是人为设计的一组计算任务，不包含实际应用中的关键算法，它的测试结果与现实应用的性能关联度较低。

CoreMark 提供更真实的工作负载（采用了一些在嵌入式系统中常见的计算任务）：

1、矩阵运算（Matrix Manipulation）

  • 作用：测试 CPU 处理数学运算的能力（如加法、乘法）。
  • 现实应用：图像处理、信号处理、科学计算。

2、链表操作（Linked List Manipulation）

  • 作用：测试指针操作和内存管理能力。
  • 现实应用：操作系统、数据库、动态数据结构（如哈希表、图）

3、状态机操作（State Machine Operation）

  • 作用：测试 CPU 在不同条件下执行不同代码路径的能力（数据相关的分支）。
  • 现实应用：协议解析（如 TCP/IP 协议栈）、控制系统（如电梯控制、嵌入式控制器）。

4、循环冗余校验（CRC，Cyclic Redundancy Check）

  • 作用：测试 CPU 计算数据校验码的能力。
  • 现实应用：网络通信（数据包校验）、存储设备（数据完整性检查）。

编译器优化

Dhrystone 的问题：容易被编译器优化

  • Dhrystone 的代码结构简单，其中许多计算的结果在编译时就可以被确定（称为 compile-time constants）。
  • 现代编译器（如 GCC、LLVM）在优化时，会尝试消除不必要的计算（称为 dead code elimination）。

Dhrystone程序容易受到编译器的影响，当这些发生的时候，其结果就不可相信；

CoreMark 会避免被编译器优化：

  • CoreMark需要计算的数值在编译时无法获取，只有在运行时才能得到；

  • 因此编译器虽然会进行一定程度的优化，但是无法在编译阶段预计算结果，会强制让 CPU 运行完整的计算过程

计时部分库函数的调用

Dhrystone 的问题:

  • Dhrystone 在计时阶段 调用了 标准库函数（如 printf()、strcpy() 等）

  • 库函数的执行时间** 依赖于 编译器和库实现，不同平台上的 库函数性能可能差异很大。

不同系统的 Dhrystone 分数可能受库函数影响，而不是 CPU 的真实计算能力

CoreMark 的改进：不在计时部分调用库函数

  • CoreMark 设计时，所有计算都由基准测试代码自己实现，不依赖外部库函数。
  • 这样可以保证所有平台运行完全相同的计算工作负载，不会因库实现不同而影响结果。

✅ 示例：Dhrystone 受库函数影响

➡ 问题：

  • strcpy() 的执行效率取决于 所用 C 标准库的实现，不同编译器可能优化程度不同。
  • 测的可能是库的性能，而不是 CPU 本身的计算能力。

✅ 示例：CoreMark 避免库函数

➡ 优点：

  • 确保所有 CPU 运行相同的代码，不受不同库函数实现的影响。
  • 测试 CPU 计算能力，而不是测试库的优化程度。

版本问题

Dhrystone没有一个固定的官方版本；

CoreMark有官方版本：http://www.coremark.org/

  • ANSI C 兼容性

ANSI C（American National Standards Institute C）指的是标准化的 C 语言。

CoreMark 代码严格遵循 ANSI C 标准，确保在所有符合 ANSI C 的编译器上都能正确编译和运行。

  • ANSI C（C89/C90）：1989 年标准化，保证代码可以在不同编译器和系统上运行。
  • 非 ANSI C 代码 可能会使用 编译器特定的语法，导致在某些平台上无法编译或运行。

Dhrystone 的最早版本在 ANSI C 标准发布前编写，使用了一些 非标准 C 语法

CoreMark 代码严格遵循 ANSI C 标准，确保在所有符合 ANSI C 的编译器上都能正确编译和运行。

结果格式不统一：Dhrystones per second, DMIPS, or DMIPS/MHz

运行CoreMark

CoreMark是一个小的benchmark程序，必须要运行多次才能获得可复制的分数

  1. 减少处理器状态不一致带来的影响
  • 问题描述：
    处理器的状态（如缓存状态、分支预测状态等）可能会影响测试结果。如果每次运行 CoreMark 时处理器的初始状态不一致，测试结果可能会有较大波动。
  • 解决方法：
    ARM 建议按照以下步骤运行 CoreMark：
    1. 两次验证运行（Validation Runs）：
      在正式测试之前，先运行两次 CoreMark，以确保测试环境正确设置，并且处理器状态趋于稳定。
    2. 至少十次性能运行（Profile Runs）：
      在验证运行之后，至少运行十次 CoreMark，以收集足够的数据。
    3. 计算平均值：
      对十次性能运行的结果取平均值，作为最终的测试结果。
  • 目的：
    通过多次运行和取平均值，可以减少处理器状态不一致带来的波动，提高测试结果的可靠性。
  1. 最小执行时间要求
  • CoreMark 要求：
    CoreMark 的官方要求是，每次运行的执行时间至少为 10 秒，以确保测试结果的准确性。
  • ARM 建议：
    ARM 建议将最小执行时间延长至 30 秒，以进一步减少测试结果的波动性。
  • 原因：
    • 较长的执行时间可以平滑短时间内的性能波动（如中断、缓存未命中等）。
    • 较长的执行时间还可以更好地反映处理器的持续性能。
  1. 执行时间与处理器频率的关系
  • 执行时间的影响因素：
    CoreMark 的执行时间取决于处理器的频率和性能。频率越高，执行时间越短；频率越低，执行时间越长。
  • 示例：
    ARM 提供了一个参考示例：
    • 在 Cortex-M4 处理器 上，运行频率为 168 MHz 时，15000 次迭代 的执行时间约为 30 秒。
  • 如何调整迭代次数：
    如果处理器的频率不同，可以通过调整迭代次数来控制执行时间。例如：
    • 如果处理器频率较低，可以增加迭代次数以达到 30 秒的执行时间。
    • 如果处理器频率较高，可以减少迭代次数以避免执行时间过长。

QSPI

发表于 2025-01-21 分类于 SoC设计 Valine：

Quad SPI接口（QSPI）是一种同步串行数据链路，在主模式下提供与外部设备的通信。它类似于SPI协议，只是它有额外的数据线。

普通SPI有四条通信线路：芯片选择、时钟、MOSI和MISO。对于QSPI，可提供额外的数据线。因此，命令/数据/地址是根据所选模式通过单、四或双IO发送的。由于数据是通过多条线路发送的，与标准SPI传输相比，它有助于增加带宽。

Dhrystone

Dhrystone 是一种综合性基准测试，由 Reinhold P. Weicker 于 1980 年代早期开发，旨在客观地测量和比较微处理器的性能。当时，微处理器行业发展迅速，不同厂商之间竞争激烈，各自声称其产品性能更优。为了提供一种统一的、客观的性能评估方法，Dhrystone 应运而生。

Dhrystone 的主要特点包括：

  1. 综合性测试：通过模拟的工作负载测试处理器的计算性能，而非依赖实际应用程序。
  2. 使用 C 语言编写：C 语言在当时被广泛支持，具有良好的可移植性和跨平台能力。
  3. 简单易用：Dhrystone 设计简洁，易于在各种微处理器上实现和执行。

Dhrystone 的简单性和可移植性使其成为当时开发者和用户的热门选择，并广泛用于评估微处理器的性能。

CoreMark

CoreMark 是一种基准测试工具，于 2000 年代早期开发，旨在适应微控制器领域的不断变化。相比早期的 Dhrystone 基准测试，CoreMark 针对更复杂、更强大的微控制器进行了优化，特别是为了评估多线程和多核处理器的性能。

CoreMark 的主要特点包括：

  1. 支持多线程和多核处理器：CoreMark 提供了专门的测试场景，能够评估处理器的并行处理效率和性能扩展能力，适应当时多核架构的趋势。
  2. 广泛的操作范围：CoreMark 包含整数运算、位操作、控制流和内存访问等多种操作，模拟嵌入式系统和实时应用中的典型任务。
  3. 更贴近实际应用：通过更复杂的内存访问模式和多样化的操作，CoreMark 更能反映处理器在嵌入式和实时应用领域的实际性能。

CoreMark 的引入使得处理器性能评估更加全面和准确，尤其在多核并行计算能力和嵌入式系统性能优化方面具有重要意义。

Dhrystone和CoreMark的区别

基准测试在被测试的微控制器上执行，其结果通过测量微控制器执行指令所需的时间来评估性能。以下是 Dhrystone 和 CoreMark 的区别和特点：

Dhrystone

  • 评估方式

    ：通过执行一系列操作来测量性能，包括：

    • 字符串操作
    • 整数算术运算（如加法、减法）
    • 位移操作
    • 逻辑运算

  • 特点：适合评估基本处理器性能，但测试内容相对简单，更偏向整数运算。

CoreMark

  • 评估方式

    ：使用多种复杂的工作负载来测量性能，包括：

    • 矩阵操作（Matrix Manipulation）
    • 链表操作（Linked List Manipulation）
    • 状态机仿真（State Machine Emulation）

  • 特点：测试更全面，能反映处理器在复杂任务中的性能，特别是在嵌入式和实时系统中的实际表现。

总结

  • Dhrystone 偏向于基本性能的测量，适合单线程、简单操作的处理器。
  • CoreMark 提供更复杂的测试场景，更适用于评估现代微控制器的多任务处理能力和整体性能。

基准测试的局限性

尽管基准测试（如 Dhrystone 和 CoreMark）在评估微控制器性能方面提供了有价值的见解，但它们存在一些基本的局限性，需要在解读结果时加以注意：

主要问题：

  1. 编译器优化
    • 编译器的优化选项、内联决策等设置会显著影响基准测试结果。
    • 不同的编译器和配置可能使结果偏离实际应用性能。
  2. 性能侧重点
    • 基准测试通常聚焦于微控制器的某些特定方面，如整数运算、控制流和内存访问。
    • 然而，这些测试无法涵盖实际应用中所有影响性能的因素（如浮点运算、I/O 性能等）。
  3. 代码规模有限
    • 基准测试的代码规模较小，通常仅测试处理器缓存的性能，而对内存层次结构（如 DRAM）影响有限。
    • 因此，无法全面反映复杂应用的真实性能。

推荐方法：

  • 补充真实应用分析
    结合基准测试结果与实际应用的性能分析，评估微控制器在具体场景中的表现。
  • 使用领域特定的基准测试
    针对目标应用领域（如嵌入式、汽车电子）的特定基准测试，提供更相关的性能数据。
  • 综合性能评估技术
    借助多种评估方法（如实时调试和运行时分析），获得微控制器性能的全面理解。

libc.so.6丢失

libc.so.6是一个动态链接库文件，是GNU C Library的一个共享函数库

可以这么说，Ubuntu下有个问题，好多文件都在/lib/x86_64-linux-gnu/下，弄得很麻烦，需要做软连接才可以使用。更巧的是，这个libc.so.6需要手动链接一下。

知道咋回事了，就开始做吧：

1、查找libc.so.6文件的具体路径

通常情况下Ubuntu下是这样的：

32bit: /lib/i386-linux-gnu/libc.so.6

64bit: /lib/x86_64-linux-gnu/libc.so.6

此外使用命令查看：

#dpkg -L libc6

2、软链接

ln -s /lib/x86_64-linux-gnu/libc.so.6 /lib/libc.so.6
