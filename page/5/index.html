  • 静态计算：在编译时确定结果，不会在运行时计算。
  • 适用对象：
    • 基本数据类型（如 int、char）。
    • 数组（返回整个数组的字节数）。
    • 指针（返回指针本身的字节数，而非指向的内容）。
    • 结构体或联合体。
    • 无法计算malloc，动态分配内存的大小。

对于C语言来说：

  • \0是一个特殊字符；在存储中的数值为0；字符串常量是以\0结尾；%s输出，以\0为结束；
  • C只是简单地将字符串储存为字符数组，并且在结尾加上\0（空字符）

思考如下问题:

上面两种结果，sizeof输出的结果符合预期，但是%s的输出怎么解释？
🔍 C标准:
C标准对字符串的定义如下（C99 §6.4.5）：

A string literal is a sequence of characters enclosed in double quotation marks, optionally containing escape sequences, and terminated by the null character (\0).

而在使用 %s 的时候，C标准规定（C17 §7.21.6.1）：

If no precision is specified, the array shall contain a null character, and the characters are written up to (but not including) the terminating null character.

==》如果字符数组没有以 \0 结尾，printf 会继续读取内存，直到偶然遇到一个 \0，这可能导致程序崩溃、输出乱码或其他不可预测的行为

原因 1：内存布局的巧合

  • 在你的程序中，full_name 数组之后的内存恰好是 \0。
  • 例如，编译器可能在 full_name 数组之后分配了一块初始化为 0 的内存，或者 full_name 数组之后的内存恰好是 \0。
  • 这种情况下，printf 会“幸运地”遇到 \0，从而正确终止输出。

原因 3：操作系统的内存管理

  • 操作系统在分配内存时，可能会将新分配的内存初始化为 0（尤其是在调试环境中）。
  • 例如，Linux 的 glibc 在某些情况下会将堆内存初始化为 0。

NULL的定义：

C 的NULL定义：#define NULL ((void *)0)

  • 0：
    • 在 C 语言中，0 是一个整数类型的数值
  • (void *)：
    • void * 是一种通用指针类型，可以指向任何类型的数据。
  • (void *)0：
    • 将整数 0 转换为 void * 类型的空指针。
    • 空指针表示：不指向任何有效内存地址的指针
    • 这种表示方式明确告诉编译器，NULL 是一个指针，而不是整数

为什么需要 (void *)0？

  • 类型安全：
    • 在 C 语言中，指针和整数是不同的类型。虽然 0 可以表示空指针，但直接使用 0 可能会导致类型不明确。
    • 使用 (void *)0 可以明确表示 NULL 是一个指针。
  • 通用性：
    • void * 是一种通用指针类型，可以隐式转换为任何其他指针类型。
    • 因此，(void *)0 可以安全地用于所有指针类型。

C++的NULL定义：#define NULL 0

在 C 语言中，void * 是一个“通用指针”，几乎可以隐式转换为其他任意指针类型。

但是C++强调类型安全，void * 并不能直接指向其他指针类型

因此C++标准中添加了特性：整数 0被解释为空指针常量

C++11新标准NULL

引入了专用的空指针常量nullptr，解决0是整数还是指针的二义性问题；

DESCRIPTION

man is the system’s manual pager.

Each page argument given to man is normally the name of a program, utility or function. The manual page associated with each of these arguments is then found and displayed.

手册章节

  • A section, if provided, will direct man to look only in that section of the manual.
  • The default action is to search in all of the available sections following a pre-defined order (see DEFAULTS), and to show only the first page found, even if page exists in several sections.

The table below shows the section numbers of the manual followed by the types of pages they contain.

1 Executable programs or shell commands
2 System calls (functions provided by the kernel)
3 Library calls (functions within program libraries)
4 Special files (usually found in /dev)
5 File formats and conventions, e.g. /etc/passwd
6 Games
7 Miscellaneous (including macro packages and conventions), e.g. man(7), groff(7), man-pages(7)
8 System administration commands (usually only for root)
9 Kernel routines [Non standard]

page章节

A manual page consists of several sections.

Conventional section names include NAME, SYNOPSIS, CONFIGURATION, DESCRIPTION, OPTIONS, EXIT STATUS, RETURN VALUE, ERRORS, ENVIRONMENT, FILES, VERSIONS, CONFORMING TO, NOTES, BUGS, EXAMPLE, AUTHORS, and SEE ALSO.

shell命令的单引号和双引号

  • 单引号

当你使用单引号包裹字符串时，Shell 会严格地保留其中的字符原样，这意味着不对其中的任何特殊字符进行扩展或替换。特别是其中的 $、*、? 等特殊字符也不会被展开。

  • 双引号

当你使用双引号包裹字符串时，Shell 会对其中的一些特殊字符（如 $、反引号 等）进行扩展，但是对于 \*、? 和 [] 这些字符，Shell 会 进行匹配，而不会被“保留原样”。

ls/find/grep匹配

ls和find是使用通配符进行匹配，但是ls和find运行时对通配符解释的时机不同；

grep是使用正则表达式进行匹配

假设有三个文件：file1.txt file2.txt file3.log

📌 ls *.txt

这个命令运行时，其实是 shell 做了这件事：

ls 程序根本看不到 *.txt，它接收到的是两个具体文件名，是 shell 提前帮你匹配好了。

📌 find . -name '*.txt'

这个命令运行时，shell只是把 *.txt 这个字符串原样传给 find，find 自己再去查：

find 内部才去判断哪些文件名和 '*.txt' 匹配，它使用自己的匹配机制（不是正则，是 shell-style 匹配）。

NOTE：find . -name *.txt

不加引号，shell 会先把 *.txt 展开成 file1.txt file2.txt

📌 grep 'D*p' file3.log

这个命令运行时，shell只是把 a*c 这个字符串原样传给 grep，让grep自己再去做正则匹配；

通常加引号，防止正则符号被 shell 误解释

虚拟机以及虚拟磁盘

  1. 虚拟机配置文件
  • 文件格式：.vmx 文件
  • 作用：
    • 这是一个文本文件，用于存储虚拟机的配置信息，例如硬件设置（内存大小、CPU 数量、网络配置等）、虚拟设备（如 CD/DVD 驱动器、USB 控制器等）以及其他虚拟机相关的参数。
    • 它是虚拟机的“元数据”文件，VMware 通过读取这个文件来加载虚拟机的配置。
  • 示例文件名：Ubuntu.vmx
  1. 虚拟磁盘文件

  • 文件格式：.vmdk 文件

  • 作用：

    • 这是虚拟机的虚拟磁盘文件，用于存储虚拟机的操作系统、应用程序和用户数据。
    • 它模拟了一个物理硬盘，Ubuntu 系统会将其视为一个真实的磁盘设备，并在其中安装和运行。
    • .vmdk 文件可以动态扩展（thin provisioned）或预先分配全部空间（thick provisioned），具体取决于创建虚拟机时的设置。

  • 示例文件名：Ubuntu.vmdk

  • .vmx 文件：虚拟机的配置文件，定义虚拟机的硬件和设置。

  • .vmdk 文件：虚拟磁盘文件，存储虚拟机的操作系统和数据。

这两个文件是 VMware 虚拟机的核心组成部分，缺一不可。如果需要迁移或备份虚拟机，通常需要同时复制这两个文件。

  • BISO模式：UEFI
  • 磁盘分区形式：GUID分区表（GPT）
  • 关闭BitLocker
  • 关闭独显直连，独显和核显
  • 压缩Windos分区，从硬盘2压缩出512GB
  • 进入BIOS：关闭secure boot，关闭Inter RST，U盘设置为优先启动项
  • 重新启动，U盘引导进入Ubuntu安装

详细步骤

1、首先使用U盘制作启动盘

2、压缩磁盘512GB空间，未分配

3、ubuntu安装时候的内存分区：

  1. /：根目录分区，大小规划204800M 200GB
  2. /boot：boot分区，大小规划512M 0.5GB
  3. /swap：swap分区，大小规划16*1024=16384M 16GB
  4. /home分区：

决定不单独进行分配空间，只分配/根目录分区

/ 和 /home分区之间的关系：

  1. 默认情况下：/ 分区包括 /home
  • / 分区
    • / 是 Linux 文件系统的根目录，所有其他目录（包括 /home）都是它的子目录。
    • 如果系统没有单独为 /home 分区，那么 /home 目录会存储在 / 分区中。
  • 示例
    • 假设 / 分区大小为 50GB，那么 /home 目录会占用 / 分区的一部分空间。
  1. 单独为 /home 分区
  • /home 分区
    • 如果系统为 /home 单独分配了一个分区，那么 /home 目录将不再属于 / 分区，而是独立存储在自己的分区中。
    • 这种设置的好处是，即使系统重装或 / 分区损坏，/home 分区中的数据仍然可以保留。
  • 示例
    • 假设 / 分区大小为 20GB，/home 分区大小为 100GB，那么 /home 目录会占用独立的 100GB 空间。

ARM Dhrystone

发表于 2025-03-06 分类于 ISA和处理器 ， 处理器性能指标和计算单元 Valine：

简介

第一个版本的Dhrystone benchmark 是1984年开发，主要是用来测试整数运算的性能；

1988年，Dhrystone有第二个版本V2，支持Ada, C and Pascal languages

==》之前流行的原因：

  • 代码非常简单
  • 编译和运行快
  • 比 MIPS 更准确地衡量 CPU 性能

==》现在，Dhrystone已经不在是代表性的基准测试，因为局限性非常明显：

  • 编译器优化问题
  • 不能代表真实应用，只能测试整数运算，无法测试浮点计算、向量运算、多线程等
  • result report形式不统一：Dhrystones per second、DMIPS、DMIPS/MHz

==》新基准测试

CoreMark

✅ 代码优化难度更大，编译器无法轻易优化掉计算，测试 CPU 实际运算能力

SPEC（Standard Performance Evaluation Corporation）

✅ 涵盖整数、浮点、存储、内存等多个领域，测试更全面，适用于高端CPU

ARM使用的是Dhrystone 2.1 的C语言版本

结果计算公式：Dhrystones per second = number of runs / execution time

为了结果的有效性，ARM推荐运行至少20秒；

编译选项

不允许使用如下编译选项：

• function inlining
• multifile compilation

Library functions required by Dhrystone

  • scanf() and printf() to read the iteration count input by the user and to report back the benchmarking results
  • clock() to compute the execution time of the benchmark
  • strcpy() and strcmp() to perform string assignment and comparison

上述函数在如下两种库中均支持：

  • standard C library ，included with the ARM Compiler toolchain and the Keil™ MDK
  • microlib （size-optimized C library ），included with the Keil MDK toolkit.

Command-line options

Speed-optimized

Size-optimized

Running Dhrystone

Dhrystone is a very small integer benchmark that must be run multiple times in order to obtain
reproducible numbers. ARM recommends you run Dhrystone ten times with varying iteration
counts and that each run takes at least 20 seconds.

In order to minimize the variation caused by inconsistent processor states, ARM recommends
you disregard the first run and calculate the average for the remaining nine runs. In addition,
when running Dhrystone on a bare-metal system, before each run you must initialize the
processor by resetting, clearing and enabling the caches, TLBs, BTBs and other
micro-architectural features such as branch prediction.

Measurement characteristics

DMIPS = Dhrystones per second / 1757

A more commonly reported figure is DMIPS / MHz
