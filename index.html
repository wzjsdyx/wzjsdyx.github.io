Makefile使用方法

发表于 2025-09-25 更新于 2025-09-28 分类于 一生一芯计划 ， E阶段 Valine：

make执行及参数传递

make 用来根据 Makefile 中的规则，自动化执行构建任务

-C /path/to/this_repo:表示先切换到指定目录，再执行 Makefile

Makefile中的默认目标是什么？

有默认的名称吗？例如all或者default?

Makefile中的默认目标是第一个目标，没有默认的名字；make执行的就是第一个目标；

但是通常的Makefile脚本中，有些名字是约定俗成的：

  • all：代表编译所有东西（最常见，约定俗成）

  • default：有些项目用这个表示默认目标

  • install：安装软件

  • clean：清理文件

Makefile中的变量赋值及调用

  • = 普通赋值，每次展开时重新计算

FOO = $(BAR)
BAR = hello

👉 延迟展开：右边内容不会立刻求值，而是等到用到 $(FOO) 时再去展开。

  • := 立即展开赋值（只计算一次）

FOO := $(BAR)
BAR = hello

👉 立即展开：右边的内容在赋值那一刻就求值，以后再改 BAR 也不会影响 FOO。

  • ?= 条件赋值：只有当变量之前没有定义过时才会赋值

FOO ?= hello

👉如果之前没定义过 FOO(即没有普通赋值定义或者命令行传入)，就赋值为 hello；如果用户或前面定义过，则保持原值。

很常见于跨平台项目，目的是给项目设定 默认值，但又允许用户在命令行覆盖

  • += 追加

变量追加

  • 使用$(VAR)引用变量

特殊符号

  • $@ → 当前目标

  • $^ → 所有依赖

  • $< → 第一个依赖

Makefile的内置函数

表达式 作用 例子 输出
$(shell cmd) 执行 shell 命令并返回结果 $(shell echo hi) hi
$(abspath path) 转换成绝对路径 $(abspath ./src) /home/user/project/src

input.vc

这是几条和 文件搜索路径 / 库解析 有关的选项。

(1) +librescan

  • 意思是：允许重复搜索库文件。
  • 默认情况下，Verilog 编译器（包括 Verilator）在 -y 指定的目录下找到一个匹配的模块文件后就停止，不再继续找同名模块。
  • 加上 +librescan，会继续扫描目录中剩下的文件，避免因为库里有多个定义而漏掉。
    👉 一般用于 大 IP 库目录，确保所有依赖模块都能找到。

(2) +libext+.v+.sv+.vh+.svh

  • 指定 允许的文件扩展名。
  • +libext+.v+.sv+.vh+.svh 表示依次尝试：
    • .v (Verilog)
    • .sv (SystemVerilog)
    • .vh (Verilog header)
    • .svh (SystemVerilog header)
  • 当在 -y 指定的目录中搜索某个模块时，编译器会自动尝试这些扩展名。
    👉 这样就不用手动写出每个文件名，模块引用时只要写 module_name，工具会自动补齐后缀。

(3) -y .

  • 指定库搜索路径，. 表示当前目录。
  • 当 Verilator 找不到某个 module xxx 的定义时，就会去 . 目录下扫描所有符合 +libext 的文件。
    👉 常用来放 IP 库或者模块集合。

最终形成的效果：

综合起来，input.vc 的意思是：

  1. 当仿真编译时，如果遇到未定义的模块，Verilator 会去 当前目录 (.) 搜索。
  2. 搜索时，会尝试 .v / .sv / .vh / .svh 这些扩展名。
  3. 即使找到一个匹配文件，也会继续扫描，避免漏掉别的依赖。

Makefile_obj

Makefile_obj 的作用是 给 Verilator 生成的核心规则 (Vtop.mk) 提供一个编译环境。

核心逻辑在 Vtop.mk，而 Makefile_obj 只是：

  1. 设置默认目标 (Vtop)；
  2. 定义编译命令和依赖生成规则；
  3. 启用调试宏和优化选项；
  4. 自动处理头文件依赖。

所以它相当于一个 薄包装层，让你在 obj_dir 下敲 make 就能编译，而不用手动关心一堆参数。
