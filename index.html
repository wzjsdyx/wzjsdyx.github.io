AXIM接口的属性配置

关于AXI协议的几个理解：

  • outstanding，Outstanding 指主设备在未收到之前某个事务的响应之前，可以继续发出的新事务的数量；当前的transaction可以不用等之前的transaction就可以发送；

  • out-of-order，主设备可以同时发出多个不同 ID 的事务，从设备可以按任意顺序返回它们，但要求 相同 ID 的事务必须按顺序完成（以保证同一条流内的顺序性）。不同 ID 之间可以乱序；

如果transaction有着不同的ID，那么不同顺序的transaction可以不按照不同顺序返回;

如果transaction有着相同的ID，那么相同顺序的transaction需要按照发送顺序返回;

例如：先不考虑interleaving 这个features，如果我发送读请求，按照先后顺序，先trasA-ID0,然后transB-ID1,最后transC-ID1,那么返回的时候，会有几种可能的情况？只要保证transB再transC之前就行，因为都是ID1；因此约束条件是：事务B必须在事务C之前返回，而事务A（ID0）可以与它们任意穿插。

可能的返回顺序（不考虑交织，即每个事务的数据连续返回）共有以下3种：

  1. A, B, C：完全按发起顺序返回。
  2. B, A, C：ID1的B先返回，然后ID0的A返回，最后ID1的C返回。
  3. B, C, A：ID1的两个连续返回，然后ID0的A返回。
  • interleaving，Interleaving 指在同一个从设备返回的多个读事务中，数据可以以 更细的粒度（数据节拍） 交叉传输。即，不同事务的数据节拍可以交替出现在读数据总线上。

AXI3支持读和写interleaving；

AXI4和AXI5不支持写interleaving；

功能安全基础知识

发表于 2026-01-23 分类于 车规MCU Valine：

共因失效

车规芯片功能安全中的 “共因失效”，指的是由一个共同的单一原因或事件，导致芯片内多个原本独立的冗余单元、安全机制或不同功能模块同时或相继失效，从而使系统性的安全防护完全崩溃的情况

在双核锁步中，让一个核心延迟（“打拍子”）的核心目的，是为了将可能同时影响两个核心的“共因瞬态故障”，转化为一个能被比较器捕捉到的“瞬时差异”

打拍操作的根本目的：制造时间差

  • 如果没有延迟，两个核心在同一时钟沿同步工作，那么一个短时间的电压毛刺或电磁干扰很可能在同一时刻被两个核心的触发器采样到，导致它们犯下完全相同的错误。比较器会发现输出一致，从而无法检测到这个实际发生的故障，系统会带着错误继续运行，这是安全设计无法接受的

  • 通过强制让核心B延迟3个时钟周期，相当于在两个核心的运算路径上设置了一个“时间屏障”。这使得：任何持续时间小于3个周期的瞬态干扰，几乎不可能在同一运算阶段同时影响两个核心

对于永久的硬件故障,无法发出报警

同步异常和异步异常

发表于 2025-11-27 分类于 ARM ， ARMv8-A架构 Valine：

🔍 同步异常 vs 异步异常 —— 为什么影响不同？

🟢 同步异常（如：除0、访问非法地址、指令未定义）

  • 异常一定是由当前指令触发
  • 触发点 精确对齐某条指令
  • 因此 CPU 必须把状态 整理成“这条指令之前全部完成，这条指令之后全部未执行”

👉 所以在同步异常发生前：

  • CPU 会强制清空写缓冲
  • 完成必要的 cache 写回
  • 禁止乱序对外可见的影响
  • 最终达到 精确异常（precise exception）语义

所以虽然内部有乱序 / 缓冲 / cache，但 CPU 会为了精确异常而把状态整理干净，然后异常可恢复。

🔴 异步异常（如：外部中断、DMA报警、定时器）

  • 异常不是由当前指令触发
  • 触发点和正在执行的指令无关系
  • CPU 不能提前预测在哪里需要保存状态

👉 异步异常发生时可能出现：

状态 同步异常 异步异常
重排已完成？ 会等到完成 可能未完成
写缓冲是否已清空？ 会强制清空 可能还积累
cache 脏数据是否写回？ 必须写回 可能未写回
程序可恢复性 高 低

因此，异步异常发生时，系统可能看到：

  • 有的内存写已完成，有的未完成
  • 顺序不符合程序语义
  • 甚至可能回滚掉 cache 写入的值（write-back 冲刷）

继续执行会产生不可预测错误，因此异步异常通常不可恢复，处理方式往往是：

  • 调度到中断服务程序，
  • 不直接返回到引发点（除非架构定义了精确中断机制，例如 ARM GIC 支持 IRQ/FIQ 的异常返回）
